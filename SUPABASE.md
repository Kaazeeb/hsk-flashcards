# Supabase documentation

This app uses Supabase for authentication plus remote progress/visibility sync. The app is a browser-only static app, so the public anon key can be visible in client code. Data privacy must be enforced by Row Level Security.

`supabase_starter.sql` is the executable starter schema for a new project. This document explains how the current code uses that schema.

## Client configuration

`js/auth.js` creates a Supabase client with:

- URL/key constants embedded in the browser bundle.
- `auth.persistSession: true`
- `auth.autoRefreshToken: true`
- `auth.detectSessionInUrl: true`

Auth calls used by the app:

- `auth.getSession()` on startup.
- `auth.onAuthStateChange(...)` to update UI state.
- `auth.signUp({ email, password, options: { emailRedirectTo } })`.
- `auth.signInWithPassword({ email, password })`.
- `auth.signOut()`.

## Tables

### `profiles`

Purpose: lightweight profile row linked to each authenticated user.

Columns:

- `id uuid primary key references auth.users(id) on delete cascade`
- `email text`
- `display_name text`
- `created_at timestamptz default now()`
- `updated_at timestamptz default now()`

A trigger on `auth.users` inserts/updates `profiles` after user creation.

RLS:

- Authenticated users can select, insert, and update only their own profile where `auth.uid() = id`.

### `app_review_events`

Purpose: append-only event stream for vocabulary progress, vocabulary Smart reviews, image progress, sentence/study Smart reviews, and reset epochs.

Columns:

- `user_id uuid not null references auth.users(id) on delete cascade`
- `event_id text not null`
- `kind text not null`
- `card_id text not null`
- `set_id text`
- `payload jsonb not null default '{}'::jsonb`
- `occurred_at timestamptz not null default now()`
- `created_at timestamptz not null default now()`
- Primary key: `(user_id, event_id)`

Current event kinds:

- `learn_seen`
- `practice_translation`
- `practice_pinyin`
- `smart_fsrs`
- `image_learn_seen`
- `image_smart_fsrs`
- `sentence_smart_fsrs`
- `review_reset`

Payload patterns:

- Practice deltas: `{ "shown_delta": number, "correct_delta": number, "wrong_delta": number, "epochId"?: string }`
- Smart reviews: `{ "rating": 1|2|3|4, "epochId"?: string }`
- Reset epoch: `{ "epochId": string, "reason": string }`

Card id rules:

- Vocabulary events store compact refs, for example `idx:123`.
- Sentence and image events store their catalog card ids directly.
- Reset rows use `card_id = "__review_epoch__"`.

RLS:

- Authenticated users can select and insert only their own rows where `auth.uid() = user_id`.
- No update/delete policy is provided for normal app use; review events are append-only.

Important indexes in the starter SQL cover user/kind, user/card, user/set/card, chronological replay, epoch-filtered replay, and sentence/image Smart review reads.

### `app_card_visibility_bits`

Purpose: compact per-user, per-deck, per-mode visibility state for Setup.

Columns:

- `u uuid not null references auth.users(id) on delete cascade`: owner user id.
- `d integer not null`: stable deck key generated by `js/visibility-bitset.js`.
- `m smallint not null`: mode key, `0=learn`, `1=practice`.
- `z boolean not null default true`: default visibility for the row.
- `n integer not null default 0`: card count represented by the bitset.
- `x text not null default ''`: base64 exception bitset.
- `t timestamptz not null default now()`: last update time.
- Primary key: `(u, d, m)`.

RLS:

- Authenticated users can select, insert, update, and delete only rows where `auth.uid() = u`.

## Read flow

`js/auth.js` exposes a remote adapter through `auth.getRemoteAdapter()`.

On `loadAppData()` the adapter reads:

1. `app_review_events` using paginated `.select(...).range(...)` calls.
2. `app_card_visibility_bits` using `.select("d,m,z,n,x,t").eq("u", userId)`.

Review event replay is epoch-aware:

- The app first reads `review_reset` rows.
- If a latest reset with `payload.epochId` exists, it then reads only non-reset events whose `payload->>epochId` matches that active epoch.
- If there is no reset epoch, it reads the full event stream.
- Rows are replayed by `occurred_at`, then `created_at`, then `event_id`.

## Write flow

### Event writes

Progress and review writes are converted into rows for `app_review_events` and written with Supabase `.upsert(...)` using the `(user_id,event_id)` primary key as the conflict target. Duplicate event ids are ignored so replays and double-submits remain idempotent.

Event writes are chunked to stay below the app's JSON payload limits:

- Maximum JSON payload target: 60 KB.
- Maximum event rows per batch: 100.

### Visibility writes

Setup visibility changes call `saveVisibilityRows(...)`. The adapter coalesces rapid changes, then writes rows to `app_card_visibility_bits`:

- Rows with real exceptions are upserted on `(u,d,m)`.
- Rows that return to default visibility are deleted.

## Local/offline behavior

Supabase Auth may persist login session state. App/business data does not maintain a local offline replay queue. If a remote write fails, the current page session can continue in memory, but the failed change is not retried later automatically.

## Deployment checklist

1. Create or open the Supabase project.
2. Run `supabase_starter.sql` in the SQL editor.
3. Confirm RLS is enabled on `profiles`, `app_review_events`, and `app_card_visibility_bits`.
4. Confirm the anon key in `js/auth.js` belongs to the intended Supabase project.
5. Deploy the static files as-is; no build command is required.
6. Create a test user, sign in, complete a few cards, reload, and verify progress/visibility is restored.
